# 抖音单账号发布 PoC 操作手册

> 状态：2026-07-16 已使用测试账号 `douyin-test` 完成登录、Cookie 检查、一次 23 秒视频公开发布和作品列表回查。
>
> 当前结论：单账号有人值守链路已跑通；尚未证明多账号、长期运行或无人值守稳定性。

## 一、固定环境

- 执行器：`dreammis/social-auto-upload`；
- 本地路径：`research/douyin-account-publishing/social-auto-upload`；
- 固定上游提交：`d32280c98b51057915cfb61f5fec139a464b8178`；
- Python 环境：第三方仓库内 `.venv`；
- 账号别名：`douyin-test`；
- 登录态：`cookies/douyin_douyin-test.json`，仅保存在本机且被 Git 忽略；
- 本地安全补丁：`patches/social-auto-upload-poc-safety.patch`。

第三方仓库没有明确的根许可证文件，当前仅作为内部受控评估对象，不复制进商业维护代码。

## 二、安全补丁解决的问题

固定提交不能直接用于真号批量发布。本地补丁增加了：

1. 发布按钮每个任务最多点击一次；
2. 页面结果不明确时立即失败，不循环点击；
3. 短信验证后继续等待同一提交结果，不创建第二次提交；
4. 有头模式允许用户直接在浏览器输入验证码，程序不读取、不记录验证码；
5. `--desc` 正文会真实写入编辑器，再追加话题；
6. 修复 CLI 对旧参数对象缺少 `notef`、`bgm` 属性时的兼容问题；
7. 新增抖音单次提交、短信验证和正文写入测试。

每次上游代码更新后，都必须重新应用并核验补丁；不能假设选择器和安全行为仍然有效。

## 三、登录与检查

在第三方仓库中执行：

```bash
.venv/bin/sau douyin login --account douyin-test --headed
.venv/bin/sau douyin check --account douyin-test
```

只有 `check` 返回 `valid` 才能进入发布。账号密码、短信验证码和 Cookie 不得进入聊天、日志或 Git。

## 四、发布前预检

每次发布必须先确认：

- 视频能被 `ffprobe` 读取；
- 尺寸、时长和音视频编码符合平台要求；
- 账号别名、视频路径、标题、正文和标签一一对应；
- 用户已经明确授权公开发布；
- 作品列表中不存在同一任务；
- 使用 `--headed`，手机在手边以处理平台验证；
- 本次任务没有商品链接、定位或定时发布等额外变量。

CLI 没有真正的 dry-run。运行 `upload-video` 会产生外部发布动作。

## 五、发布命令

```bash
.venv/bin/sau douyin upload-video \
  --account douyin-test \
  --file "/absolute/path/to/video.mp4" \
  --title "作品标题" \
  --desc "作品正文" \
  --tags "标签1,标签2,标签3" \
  --headed \
  --debug
```

若出现短信验证，用户直接在浏览器中输入并点击验证。程序等待验证框消失后继续观察同一次发布，不要求用户把验证码交给 Agent。

## 六、发布后回查

命令返回成功仍不是最终证据。必须打开抖音创作者中心的作品管理页，至少核对：

1. 作品真实出现；
2. 状态为已发布或审核中，而不是草稿、未通过；
3. 时长、标题和标签正确；
4. 没有重复作品；
5. 平台没有异常登录、违规或风控提示。

如果 CLI 报“结果不明确”，禁止直接重发。先回查作品列表；只有确认不存在作品，才能创建新的发布任务。

## 八、当前能力边界

已经证明：

- 一个测试账号可以独立保存 Cookie；
- 登录态可以被 CLI 检查；
- 23 秒竖屏视频可完成上传、短信验证、公开发布和后台回查；
- 单次提交安全门可防止同一进程反复点击发布。

尚未证明：

- 第二次发布能完整写入正文并让 CLI 自己识别短信验证成功；
- Cookie 跨天复用稳定；
- 两个以上抖音账号不会串号；
- 无人值守、定时发布、并发发布或长期高频发布稳定；
- 页面改版后选择器仍然有效。

下一步只做低频受控复测：同一账号隔日再发一条低风险视频，重点验证正文写入、浏览器人工短信验证和 CLI 成功回执三者一致。通过后再引入第二个测试账号。
