# 抖音、小红书、快手批量账号视频发布调研

> 调研日期：2026-07-15
>
> 目标：在现有“参考视频 → 批量成片”能力之后，评估如何将批量视频发布到批量抖音、小红书和快手账号。
> 本轮边界：只做静态调研，未安装第三方项目，未登录账号，未上传或发布视频。

## 一、决策摘要

### 1. 现在不应直接开发完整发布系统

三平台发布路线已经可以确定，但“实际稳定可用”还缺少真实账号 PoC 证据。页面自动化项目的 README 能证明作者意图，不能证明当前页面、当前账号和当前风控下一定成功。

因此下一步应该是**受控发布 PoC**，而不是立即把某个开源项目嵌入现有混剪系统。

### 2. 首选候选：`dreammis/social-auto-upload`

它是目前唯一同时满足以下条件的主候选：

- 13,318 Stars，2,309 Forks，36 位 GitHub API 可见贡献者；
- 2026-07-15 仍在更新；
- 确有抖音、小红书、快手三个视频 uploader；
- 三平台都已提供 CLI；
- 三平台都已提供独立 `SKILL.md`；
- 支持按 `account` 保存不同 Cookie，支持视频、标题、描述、标签和定时发布。

但它只应被当作**三平台发布适配器 PoC**，不是成熟的批量账号中台。它仍缺少企业级审批、RBAC、幂等、任务队列、发布后回查、审计和凭证加密。

### 3. 第二候选：`hanliang97/MatrixMedia`

这个项目更接近“矩阵发布后台”的形态：

- 抖音、小红书、快手均有上传模块；
- 提供 Electron GUI、CLI、HTTP API、MCP 和 Agent Skill；
- 支持账号列表、Cookie 状态检查、定时发布和发布历史；
- GPL-2.0 许可证文件存在；
- 2026-07-13 发布 v0.10.2，近期更新很快。

但它创建于 2026-03，当前仅 204 Stars、5 位可见贡献者，项目历史太短；现有测试主要是 CLI 参数和内部流程，不能证明三平台真实发布长期稳定。它适合做第二对照组，不应立即成为生产主线。

### 4. 平台长期路线

- **抖音**：优先申请官方开放平台发布能力；浏览器自动化作为 PoC/备用通道。
- **快手**：优先核验官方 `user_video_publish` 权限的当前申请条件；浏览器自动化作为备用。
- **小红书**：本轮在官方公开 API 目录中未找到面向普通创作者的视频/笔记发布 API；短期只能采用有人值守的浏览器自动化，或向平台确认是否存在非公开合作权限。

### 5. 一句话选型

> 不选 Dify/n8n 式可视化工作流；未来建立一个 `multi-account-video-publisher` 本地 Skill，上层由我们管任务、审批、账号和日志，底层首先用 `social-auto-upload` 做三平台 PoC，同时保留抖音/快手官方 API 适配器位置。

## 二、“实际稳定可用”的判定标准

本调研不把 Star 数或 README 宣称等同于稳定性。一个项目至少要通过七层证据：

| 层级 | 证据 |
| --- | --- |
| 代码真实性 | 三平台上传模块确实存在，不是只有宣传页 |
| 接口可编排 | 有 CLI/API/MCP/Skill，不依赖人工在 GUI 中逐条操作 |
| 多账号隔离 | 每个账号独立 Cookie/Profile，可检查过期状态 |
| 任务安全 | 有预检、人工审批、幂等键、账号串行、重复发布防护 |
| 状态真实 | 区分“上传完成”“已提交”“审核中”“已公开” |
| 当期适配 | 最近仍有针对平台页面变化的实质修复，不是只更新 README |
| 真号验收 | 用测试账号经过多日、多账号、真发布和失效恢复验证 |

按这个标准，**当前没有任何一个开源项目能被直接证明为“三平台多账号生产级稳定”**。

## 三、三平台官方能力

### 3.1 抖音

抖音开放平台官方“创建视频”文档仍可访问。已核验到的关键约束：

- 先上传视频，再创建作品；
- 需要 OAuth 用户授权；
- 代替用户发布需要 `video.create.bind` 能力；
- 应用/主体需在开放平台申请相应权限；
- 接口成功表示进入平台流程，不等于立即审核通过并公开。

结论：这是抖音长期主通道，但需要主体资质、应用审核和账号授权，不是开箱即用。

### 3.2 快手

快手官方 `KwaiOpen/KwaiOpenSDK` 代码与文档展示了：

- OAuth access token/refresh token；
- `user_video_publish` scope；
- 一键创建视频；
- 开始上传、单文件/分片上传、断点续传和发布；
- 返回作品 ID，并能查询作品信息。

但该 SDK 最后推送在 2022-04，不能单凭旧 SDK 假定 2026 年权限仍对所有申请人开放。正式设计前必须在快手开放平台控制台确认当前申请条件和限额。

结论：长期值得优先官方 API，短期仍需浏览器 PoC 做备用。

### 3.3 小红书

2026-07-15 实时请求小红书开放平台公开 API 导航，返回的分类为：

- 公共 API；
- 订单、售后、商品、库存、素材中心、物流、财务 API；
- 即时零售、会员通、供货商 API。

未发现面向普通创作者的笔记/视频发布类别。这不排除小红书向特定合作方提供非公开能力，但不能把它当作公开可申请的标准发布 API。

结论：小红书是三平台中浏览器自动化依赖最强、长期稳定性最弱的一环。

## 五、关键项目深挖

### 5.1 `social-auto-upload`

#### 值得保留的能力

- `sau douyin ...`、`sau xiaohongshu ...`、`sau kuaishou ...` 命令契约已经形成；
- 三平台都支持 `login`、`check`、`upload-video`、`upload-note`；
- 每个平台都有独立 Skill、CLI 参考和故障排查文档；
- 可通过账号名映射不同 Cookie 文件；
- 小红书上传器已有单元测试；
- 抖音 2026-07 仍在修复短信二次验证流程。

#### 不能忽略的问题

1. **许可证仍未解决**
   README 声称采用 MIT，但当前根目录 `LICENSE` 链接返回 404，GitHub API 也标记 `NOASSERTION`，Issue #236 仍是打开状态。在许可证澄清前，可以做内部技术评估，不应复制源码进入商业产品或再分发。

2. **无头登录已被实际报告识别**
   Issue #224 记录了抖音/快手 headless 登录二维码被识别并作废，以及 CLI 和旧 Web 账号数据不同步。

3. **测试不等于三平台真发布验收**
   仓库有 4 个测试文件，但没有可以证明每日用真实账号完成抖音/小红书/快手发布的公开 E2E 证据。

4. **缺少中台级状态模型**
   它可以“按账号执行一条发布命令”，但没有完整解决跨账号排期、幂等、发布后回查、审批、审计和失败熔断。

#### 定位

作为我们未来 Skill 的第一个底层适配器，不直接成为我们的账号、任务和凭证数据源。

### 5.2 `MatrixMedia`

#### 优势

- 形态最接近未来发布中台；
- `cli accounts` 可检查账号登录状态；
- `cli publish` 支持本地文件/URL、账号、标题、标签和一次性定时；
- `cli history` 提供基础发布记录；
- MCP 提供 `publish_video`、账号和历史工具；
- 小红书提供“真实 Chrome”模式，作者明确承认 Electron 内置窗口可能触发自动化警告。

#### 风险

- 仓库仅有数月历史，快速发版不等于稳定；
- 贡献者和用户量尚小；
- 小红书和快手 CLI 登录仍需先在 GUI 中完成；
- 定时任务依赖应用继续运行；
- GPL-2.0 与未来商业闭源集成方式需要单独评估；
- 仍然是浏览器自动化，无法消除页面改版和风控风险。

#### 定位

用它检验“账号列表＋历史＋CLI/MCP”是否比 `social-auto-upload` 更省自研工作，但不先做源码合并。

### 5.3 `xiaohongshu-mcp`

这是小红书单平台中社区信号最强的项目：

- 14,689 Stars、32 位可见贡献者、97 个 GitHub Release；
- 支持登录检查、图文发布、本地视频发布、搜索、详情和评论；
- 提供 HTTP API/MCP；
- 作者声称原始系统运行一年多，并在开源前稳定运营数周。

限制：

- 它不覆盖抖音和快手；
- 主 MCP 更像单账号服务，不是成熟多账号队列；
- 附带的 `post-to-xhs` Skill 有多 Chrome Profile 管理，但其主流程主要面向图文/长文，不是本项目的批量视频队列契约；
- 仓库没有 `LICENSE` 文件，不能默认可商业复用；
- README 中“每天 50 篇”属作者实操说明，不应被我们当作平台官方安全频率。

结论：可作为小红书适配层的技术对照和故障备选，不直接整仓库集成。

### 5.4 `Spreado`

仓库形态看似理想：三平台 uploader、CLI、Skill、独立 Cookie 路径、定时发布、Apache-2.0 文件、每日选择器/E2E 检查。

但 2026-07-15 最新公开报告是直接的否定证据：

| 平台 | E2E 结果 | 关键问题 |
| --- | --- | --- |
| 抖音 | 11/11 PASS | 只验证发布按钮存在，没有真正点击公开发布 |
| 小红书 | 2/7 FAIL | 已登录发布页选择器失效 |
| 快手 | 2/7 FAIL | 已登录发布页选择器失效 |

另有公开 Issue 报告登录失败。README 声称 MIT，但实际 `LICENSE` 是 Apache-2.0，说明文档一致性也需要提高。

结论：它的“自动生成平台健康报告”思路很值得借鉴，但当前版本不能作为三平台主适配器。

### 5.5 `PostBot`

它实际是浏览器扩展，适合把一份内容同步到当前已登录的多个平台页面。源码树中确实有抖音、小红书、快手的视频 publisher。

但：

- 它不是“多账号任务队列”形态；
- 仅 3 位可见贡献者，无 GitHub Release；
- package 依赖大量 `@gitcoffee/*` 包，需要另外验证供应链和开源完整性；
- 许可证在 Apache-2.0 之后加了额外限制：多租户环境需要商业授权，前端 Logo/水印/作者信息不得移除。因此不能简单当作标准 Apache-2.0 项目。

结论：适合研究“同一条内容跨平台同步”的交互，不作为我们批量账号后台的核心依赖。

## 六、建议的系统边界

不管最后选哪个 uploader，以下能力必须由我们自己掌握：

```text
现有视频生产系统
  └─ 成片 + 封面 + 各平台标题/正文/标签 + 版权/产品信息
       ↓
我们的发布任务层
  ├─ 账号注册表（平台、别名、负责人、授权状态、凭证引用）
  ├─ 视频/文案资产表（哈希、版本、来源、可用平台）
  ├─ 发布计划（一个成片→哪些平台/账号/时间）
  ├─ 内容校验和人工审批
  ├─ 幂等键（platform + account + asset_hash + schedule）
  ├─ 账号级串行、限速、熔断和人工接管
  └─ 发布状态与日志
       ↓
平台适配层
  ├─ 抖音：官方 API（长期）/ social-auto-upload（PoC/备用）
  ├─ 快手：官方 API（长期）/ social-auto-upload（PoC/备用）
  └─ 小红书：social-auto-upload（PoC）/ 专项适配器（备用）
       ↓
回查层
  └─ 已计划 → 上传中 → 已提交 → 审核中 → 已公开 / 失败 / 需人工
```

### 凭证存储

- Cookie、access token、refresh token 不进 Git、数据库明文、飞书文档或日志；
- 优先 macOS Keychain/专用 Secret Manager；
- 数据库只保存凭证引用、过期时间和最后校验结果；
- 每个账号独立浏览器 Profile/Cookie 目录，不共用登录态。

## 七、未来 Skill 应有的契约（本轮不实现）

建议名称：`multi-account-video-publisher`。

它不应只封装“点击发布”，而应对 Agent 暴露以下确定性动作：

| 动作 | 作用 | 是否允许产生外部影响 |
| --- | --- | --- |
| `accounts list/check` | 查看账号与登录状态 | 否 |
| `plan` | 将成片扩展为平台×账号发布计划 | 否 |
| `preflight` | 检查视频、文案、账号、时间和重复任务 | 否 |
| `approve` | 固化人工审批版本 | 仅写本地状态 |
| `publish --job <id>` | 只执行已审批的单个任务 | **是，必须明确授权** |
| `verify/reconcile` | 回查作品与审核状态 | 只读平台 |
| `pause-account` | 账号异常时熔断后续任务 | 停止外部动作 |

安全默认值：

- 默认 dry-run；
- 默认每个任务需要人工审批；
- 同一账号只允许一个发布任务运行；
- “发布”失败不盲目自动重试，先回查是否已生成作品；
- 登录过期、二次验证、风控或 DOM 失效时立即转人工；
- 不自动破解验证码，不伪造设备指纹，不用代理池规避平台控制。

## 八、建议的 PoC

### 8.1 PoC 目标

不验证“脚本能否打开页面”，而是验证：

1. 批量成片能否变成可审计发布计划；
2. 每个账号的登录态能否独立管理；
3. 三平台能否真实提交、公开并回查；
4. 失败后能否避免重复发布；
5. 任何账号异常时能否立即停止该账号后续任务。

### 8.2 PoC 账号和内容

- 每个平台 2 个受控测试号，共 6 个；
- 不用品牌主账号；
- 使用已审核、无敏感承诺、可公开的测试视频；
- 每个账号低频串行发布，不做同时群发；
- 发布时间和数量遵守平台当期规则与账号状态，不预设“安全群发频率”。

### 8.3 候选对照

- A 组：`social-auto-upload`；
- B 组：`MatrixMedia`；
- 小红书故障备选：`xiaohongshu-mcp` 只做运行验证，在许可证澄清前不复制源码。

### 8.4 通过门槛

一个候选只有全部满足以下条件才进入实现阶段：

1. 6 个账号的登录态完全隔离，不串号；
2. 至少 18 个真发布任务全部有完整日志；
3. 无重复发布，无标题/标签/封面串账号；
4. 可准确区分上传失败、提交成功、审核中和公开成功；
5. Cookie 过期和二次验证会转人工，不得无限重试；
6. 一个账号失败不影响其它账号；
7. 执行器关闭/网络中断后可安全恢复，不重复提交；
8. 能从平台作品列表或官方 API 回查真实结果；
9. 账号无异常登录、限制发布或处罚；
10. 许可证和商业使用边界已确认。

PoC 样本很小，即使 18/18 成功也只能证明可进入更长时间灰度，不能证明永久稳定。

## 九、不建议的做法

- 不把某个项目的 Web/GUI 整套接进现有视频生产仓库；
- 不让第三方项目成为账号、任务和日志的唯一数据源；
- 不以“命令返回 0”代替作品公开状态回查；
- 不在任务失败后直接自动再点一次发布；
- 不共享浏览器 Profile，不把 Cookie 存入 Git/飞书/网盘；
- 不追求“完全无人、无限账号、永不风控”；
- 不把 TikTok Content Posting API 当作中国大陆抖音接口；
- 不为了自动化而绕过验证码、实名认证和平台授权。

## 十、建议的下一个决策点

在实现前，只需要决定一件事：是否拿 6 个低风险测试账号（每平台 2 个）开始受控 PoC。
