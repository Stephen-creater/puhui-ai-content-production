# 小红书三账号发布 PoC 操作手册

> 状态：代码已拉取，依赖未安装，账号未登录，尚未产生任何平台发布。当前上游存在发布重试安全缺口，修复并验证前禁止真号发布。
>
> 目标：验证一台电脑能否稳定隔离 3 个小红书账号，并在人工值守下逐账号发布、回查和停止异常任务。

## 一、当前模块

本轮采用 `dreammis/social-auto-upload` 作为发布执行器：

- 本地路径：`research/douyin-account-publishing/social-auto-upload`；
- 固定上游提交：`d32280c98b51057915cfb61f5fec139a464b8178`；
- 小红书 Skill：`skills/xiaohongshu-upload/SKILL.md`；
- 命令入口：`sau xiaohongshu ...`。

第三方仓库保持独立 Git 边界。它没有明确的根许可证文件，目前只用于内部受控评估，不复制到本项目维护代码。

## 二、你需要准备什么

### 账号

准备 3 个你有权操作的真实小红书账号，并确保：

1. 每个账号都能在手机 App 正常登录；
2. 每个账号当前可以手工发布作品；
3. 实名、手机号和平台要求的验证已经完成，或者验证时你能立即处理；
4. 不使用品牌主账号做第一次测试；
5. 测试期间不要在其他电脑网页端再次登录同一个账号。

手机可以是一台，也可以是两台。手机数量不决定电脑能保存多少账号；手机主要用于扫码、短信验证和发布结果确认。

不要把密码、短信验证码或 Cookie 发给 Codex。登录必须通过你本人在小红书页面完成。

### 测试内容

优先准备 3 条不同的低风险视频，每个账号一条。每条需要：

- 本地 MP4 文件；
- 一个明确的目标账号；
- 标题；
- 正文；
- 话题标签；
- 可选封面；
- 确认可以公开发布，不含侵权素材、引流、虚假承诺或平台违禁内容。

第一轮不使用同一视频、同一标题同时铺到 3 个账号，也不测试定时发布。

### 你的时间配合

第一次测试请预留一段连续时间，并保持手机在手边。你需要处理：

- 3 次扫码或登录确认；
- 可能出现的短信、实名或安全验证；
- 每次发布前确认账号与内容；
- 每次发布后在 App 中确认作品是否真实出现。

## 三、账号别名

测试时只向 Codex提供不含隐私的本地别名，例如：

| 本地别名 | 实际账号 | 用途 |
| --- | --- | --- |
| `xhs-a` | 由你扫码确认 | 第一测试号 |
| `xhs-b` | 由你扫码确认 | 第二测试号 |
| `xhs-c` | 由你扫码确认 | 第三测试号 |

程序会生成以下本地登录态文件：

```text
cookies/xiaohongshu_xhs-a.json
cookies/xiaohongshu_xhs-b.json
cookies/xiaohongshu_xhs-c.json
```

`cookies/` 已被第三方仓库 `.gitignore` 忽略。这些文件等同账号凭证，不上传、不复制、不展示内容。

## 四、Codex执行顺序

### 阶段 0：隔离安装

Codex在第三方仓库自己的 `.venv` 中安装依赖，不写系统 Python：

```bash
cd 99_研发与临时文件/技术调研/research/douyin-account-publishing/social-auto-upload
uv venv
source .venv/bin/activate
uv pip install -e .
PLAYWRIGHT_DOWNLOAD_HOST="https://npmmirror.com/mirrors/playwright" patchright install chromium
sau xiaohongshu --help
```

完成标准：CLI 可以启动；尚未读取账号、打开登录页或发布作品。

### 阶段 0.5：发布安全门

当前固定提交的小红书发布器存在以下行为：

1. 点击“发布”；
2. 只等待 3 秒 URL 变化；
3. 页面未按预期跳转时进入无上限循环；
4. 再次点击“发布”。

因此在第一次真号发布前，Codex必须先在本地评估副本中把这段逻辑改成：

- 发布按钮最多点击一次；
- 页面未跳转时返回“结果不明确”，不继续点击；
- 记录账号别名和任务摘要，但不记录 Cookie；
- 由你在 App 作品列表回查后，再决定是否创建新任务。

修复后至少运行相关静态检查和单元测试，并保留本地补丁差异。这个安全门没有通过时，只能执行安装、`login` 和 `check`，禁止执行 `upload-video`。

### 阶段 1：逐个登录

第一次使用有头浏览器，不采用无头模式：

```bash
sau xiaohongshu login --account xhs-a --headed
sau xiaohongshu login --account xhs-b --headed
sau xiaohongshu login --account xhs-c --headed
```

每次只登录一个账号。浏览器出现登录页后，由你在手机上确认对应账号并完成扫码；Codex只检查命令结果，不接触账号密码。

### 阶段 2：只检查登录态

```bash
sau xiaohongshu check --account xhs-a
sau xiaohongshu check --account xhs-b
sau xiaohongshu check --account xhs-c
```

只有三个账号都返回 `valid`，才进入发布阶段。若其中一个失效，只重新登录该账号，不删除其他账号文件。

### 阶段 3：发布前预检

Codex先做不产生外部影响的检查：

- 视频可以被 `ffprobe` 正常读取；
- 视频路径、封面路径存在；
- 三条任务的账号、视频、标题不串号；
- 标题、正文和标签已由你确认；
- 没有相同账号加相同素材的重复任务；
- 三条任务使用有头模式并按顺序执行。

当前上游 CLI 没有真正的 `dry-run`，执行 `upload-video` 会点击发布。因此进入下一阶段前，必须同时满足：发布安全门已经通过，并且你明确说“可以发布这 3 条测试作品”。

### 阶段 4：逐条发布和回查

每次只执行一条：

```bash
sau xiaohongshu upload-video \
  --account xhs-a \
  --file "/absolute/path/to/video-a.mp4" \
  --title "测试标题A" \
  --desc "测试正文A" \
  --tags "标签1,标签2" \
  --headed
```

第一条命令结束后，不立即运行第二条。先由你在手机 App 或创作者后台确认：

1. 登录的是正确账号；
2. 作品真实出现在作品列表；
3. 视频、标题、正文和标签正确；
4. 没有重复作品、平台警告或额外验证。

确认无误后，再对 `xhs-b`、`xhs-c` 重复相同流程。

## 五、必须停止的情况

出现以下任一情况，立即停止剩余账号，不自动重试发布：

- 账号与预期不一致；
- 平台弹出风险、违规、异常登录或安全警告；
- 出现验证码、实名、短信或人工审核页面；
- 命令卡在发布按钮、超时或返回结果不明确；
- 程序说成功，但 App 中没有作品；
- 无法判断上一条是否已经提交；
- 同一作品意外出现两次。

失败后先回查作品列表，再决定是否重试；禁止直接再次点击发布。

## 六、通过标准

第一轮通过需要同时满足：

1. 三个账号分别生成独立登录态，三次 `check` 都有效；
2. 三条作品没有串账号、串视频、串标题；
3. 每条只提交一次；
4. 三条作品都能在平台作品列表确认；
5. 没有平台警告、异常登录或账号处罚；
6. 任一失败都能阻止后续自动继续。

第一轮 3/3 成功只代表单次流程跑通。下一步应连续 3 天、每天每账号最多 1 条受控测试，重点验证 Cookie 复用和页面变化；稳定后再讨论扩展到 6 个账号。

```text
手机都在手边，可以扫码和处理验证。
3 条测试视频及标题正文已经放入项目，请先安装隔离环境并只完成登录和 cookie 检查，不要发布。
```

登录态全部有效后，再单独授权发布：

```text
我已确认账号和三条内容，可以按 xhs-a → xhs-b → xhs-c 顺序逐条公开发布；每条发布后先停下来让我确认。
```
