# 项目协作规范

## 执行原则

- 保持完整任务范围、模型质量、正确性和与风险匹配的验证。
- 优先批量执行独立只读检查，复用已验证结果和固定文件路径。
- 长任务在 `work/` 保存紧凑检查点，避免重复读取大型素材。
- 原始会议、客户素材、生成中间件与最终交付物按现有目录规范存放。
- API Key 和真实凭证不得写入仓库；使用环境变量或 macOS Keychain。

## GitHub 版本管理

- 本仓库使用 `main` 作为持续交付分支。
- 每次完成一个可验证的小变更后，必须检查 `git status` 和差异范围。
- 只提交本次变更，使用简洁、可追溯的提交信息。
- 相关验证通过后立即执行 `commit` 和 `push`，不要累积多个无关改动。
- 推送失败时保留本地提交，排查网络或远端状态后继续推送，不重写已公开历史。
- 不使用 `git reset --hard`、裸 `--force` 推送或其他破坏性命令。
