# 项目协作规范

## 执行原则

- 保持完整任务范围、模型质量、正确性和与风险匹配的验证。
- 优先批量执行独立只读检查，复用已验证结果和固定文件路径。
- 长任务在 `work/` 保存紧凑检查点，避免重复读取大型素材。
- 原始会议、客户素材、生成中间件与最终交付物按现有目录规范存放。
- API Key 和真实凭证不得写入仓库；使用环境变量或 macOS Keychain。

## 存储与命名

- 顶层分工以根 `README.md` 的“目录地图”为权威：原始需求、业务内容、正式视频项目、生产工具、Agent 工具、研发归档和本地临时区不得混放。
- `work/` 是唯一的本地临时区；不得在 `99_研发与临时文件` 或业务目录内再建并行 `work/`。
- 新的项目、Skill 和脚本目录使用小写 `kebab-case`；批次、变体和分镜使用 `phaseN-vN`、`variant-NN`、`scene-NN`。
- 客户与会议原始件默认保留来源文件名；新建的内部文件使用稳定、可搜索、不含 `new`/`final2` 等临时词的名称。

## 文档洁癖

- 代码、路径、入口命令或环境变量变化时，在同一次提交中就地更新 README/AGENTS，优先修改旧说明，不叠加相互冲突的新章节。
- 每个开发里程碑或交接前使用全局 `neat-freak` Skill 执行“整理文档/同步一下/规范体检”，核对代码、文档、规则和死引用。
- 不手工修改 `~/.codex/memories/` 中的机器生成文件；项目级权威规则继续维护在本 `AGENTS.md`。
- 提交前运行 `python3 91_Agent协作工具/repository-hygiene/check_repository.py`；检查失败时先修正路径或文档，不跳过。

## GitHub 版本管理

- 本仓库使用 `main` 作为持续交付分支。
- 每次完成一个可验证的小变更后，必须检查 `git status` 和差异范围。
- 只提交本次变更，使用简洁、可追溯的提交信息。
- 相关验证通过后立即执行 `commit` 和 `push`，不要累积多个无关改动。
- 推送失败时保留本地提交，排查网络或远端状态后继续推送，不重写已公开历史。
- 不使用 `git reset --hard`、裸 `--force` 推送或其他破坏性命令。
