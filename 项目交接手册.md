# 浦汇未来 AI 短视频生产系统：项目交接手册

> 适用对象：第一次接触本项目、具备基础开发能力，需要完成项目讲解、演示或继续开发的同事。
> 仓库地址：<https://github.com/Stephen-creater/puhui-ai-content-production>
> 当前主案例：一体式预贴胶带防尘膜（Pre-taped Masking Film）产品短视频。
> 文档口径：以仓库当前 `main` 分支为准。

## 1. 先记住这一句话

本项目把“参考短视频拆解、产品事实约束、脚本变体、AI 关键帧、AI 视频、旁白字幕、质量验收和结果留档”沉淀为一组可复用的本地 Agent Skills 与确定性脚本，让同一产品能够按统一 SOP 批量生产差异化短视频，同时用人工质量门和费用安全门拦截错误素材与失控调用。

它不是 Dify、n8n 一类拖拽工作流，也不是一个需要部署到服务器的网站。它更接近一套“可执行的 AI 视频生产操作系统”：Agent 负责理解任务和编排，脚本负责稳定执行与留档，人工负责关键视觉判断。

## 2. 项目要解决什么问题

普通的 AI 视频制作往往依赖一次性对话和临时提示词，常见问题是：

- 换一个操作者就无法复刻；
- 产品结构和操作顺序容易被模型画错；
- 同一提示词反复生成，所谓“批量”只是高度相似的随机结果；
- 图片没检查就直接生成昂贵视频，错误被放大且浪费额度；
- API 超时后盲目重试，可能重复扣费；
- 成片、提示词、参数和验收结论互相脱节，无法追溯。

本项目的答案是把生产过程拆成可验证的阶段，并为每个阶段保存机器可读事实：

```text
授权参考视频 + 产品图 + 已确认产品事实
  → 参考视频拆解与模板 DNA
  → 母版脚本与受控变体
  → 独立的七维场景策略
  → 关键帧生成
  → 人工逐张验收 + 图片哈希绑定
  → 图生视频
  → 技术检查 + 1/3/5/7/9 秒视觉验收 + 视频哈希绑定
  → 旁白、字幕、后期组装
  → 20～60 秒正式成片 / 10 秒可复用场景素材
```

## 3. 当前已经能实现的功能

### 3.1 参考成片拆解

`reference-video-analyzer` 可以对经授权的参考视频提取：

- 时长、分辨率、帧率等技术信息；
- 联系表、镜头切换证据和镜头节奏；
- 口播、字幕、钩子、证明、收益点、CTA 等信息层；
- 可复用的结构模板 `template-dna.json`。

目标是学习“结构和节奏”，不是逐帧复制参考片或照搬受保护文案。

### 3.2 一条参考片裂变多个受控变体

`reference-ad-factory` 将参考片模板与产品事实分开管理，可生成不同策略的脚本。例如当前防尘膜案例已经完成：

- 清洁成本型；
- 施工效率型；
- 产品结构型；
- 专业油漆工推荐型；
- 前后对比型。

变体不是只换标题。脚本、人物、环境、镜头和视觉资产都应存在真实差异。

### 3.3 关键帧与视频批量生产

`video-batch-producer` 当前默认链路为：

| 阶段 | 当前默认方案 | 主要作用 |
| --- | --- | --- |
| 关键帧 | TokenDance `seedream-5.0-lite` | 生成 9:16 产品场景首帧 |
| 视频 | nanyao Grok `grok-imagine-video-1.5-fast` | 用合格首帧生成默认 10 秒视频素材 |
| 组装 | FFmpeg / ffprobe | 裁切、拼接、编码和技术验证 |
| 口播 | 独立 TTS 路线，当前案例使用 Alex | 保证台词、语速与字幕可控 |

Grok Fast 支持 6 秒或 10 秒；当前默认选 10 秒。供应商文档曾标注 6 秒和 10 秒同价，因此用最长时长获取更多可剪素材，但正式调用前仍须以供应商控制台的实时价格为准。

### 3.4 差异化场景策略

每条视频都必须使用独立提示词，并至少通过七个字段定义差异：

1. 物体种类；
2. 物体形态；
3. 环境场景；
4. 覆盖状态；
5. 镜头景别；
6. 镜头运动；
7. 人物/施工状态。

产品级 Excel 台账用于中文浏览、筛选和复盘；真正驱动流水线的事实源仍是 `project.json`、`keyframe-review.json` 和 `video-review.json`。重复提示词或重复七维组合会在读取 API Key 前被拒绝。

### 3.5 两层质量门

第一层是关键帧硬门。以防尘膜为例，必须同时满足：

1. 透明膜是一整张连续薄膜，无破洞、分叉或缺角；
2. 膜完整覆盖目标物；
3. 黄色胶带与膜上沿全长连续，不能悬空或超出膜；
4. 胶带贴在真实承载面，而不是错误贴到被保护物上；
5. 不得出现第二条胶带、第二张膜或多余结构；
6. 无明显变形和视觉伪影。

人工批准后，系统把结论与图片 SHA-256 绑定。图片被替换，批准自动失效；没有逐张批准，视频 API 不会被调用。

第二层是视频质量门。每条 10 秒素材固定抽取 1、3、5、7、9 秒检查。只要中途出现破膜、漏盖、断胶、额外胶带、物体变形或结构消失，整条视频即淘汰。视频结论同样绑定 SHA-256。

### 3.6 费用安全与断点续跑

- 所有付费操作先 dry-run，先显示将新增的图片数、视频数、总秒数和预计费用；
- 执行时必须显式提供 `--cost-authorized`；
- 必须设置 `--max-image-jobs`、`--max-video-jobs` 和 `--max-paid-video-seconds`；
- 已成功资产默认复用，只补缺失项；
- 可用 `--task` 定点处理单个失败镜头；
- 对提交超时或 HTTP 5xx，先查已有任务 ID，禁止直接重新提交；
- 建议首轮付费运行使用 `--retries 0`，需要重试时重新核对任务和费用后定点执行。

这套设计解决的是“批量生产时如何不失控”，而不只是“能否调用模型”。

### 4.1 已完成的正式广告批次

Phase 2 v2 已组装 5 条英文广告变体，时长分别为 20.000～21.500 秒，包含 Alex 英文旁白、逐镜字幕和独立视觉资产。历史批次的部分镜头来自 Kling，当前新生产适配器已经切换为 nanyao Grok；介绍时不要把两条链路混称为同一供应商。

对应证据：

### 4.2 已完成的场景素材验证

当前严格记录如下：

| 批次 | 严格结果 | 说明 |
| --- | ---: | --- |
| 电视完整膜返工 | 2/3 | 1 条因疑似双层膜片淘汰 |
| 饮水机完整膜返工 | 3/3 | 三条均通过五节点验收 |
| 电视差异化场景 | 2/3 | 1 条中途膜消失 |

### 4.3 当前成熟度判断

已经跑通并可复用的是：参考拆解、受控变体、七维场景设计、关键帧先审、视频后审、费用硬上限、断点续跑、哈希留证和结果归档。

仍然需要人工参与的是：产品事实确认、关键帧视觉判断、视频逐帧判断、最终广告节奏与商业效果验收。项目目前是“高约束的人机协同流水线”，不是无需监督的一键全自动工厂。

## 7. 实际跑一批视频的标准顺序

完整命令和参数以 `operator-runbook.md` 为准；操作时必须遵守下面的顺序，不可跳过图片验收直接生成视频。

### 第一步：准备输入

- 经授权的参考成片；
- 产品图片；
- 已确认的产品事实、禁用表述和操作顺序；
- 变体数量、语言、比例和费用授权边界。

防尘膜项目最重要的产品事实是：透明 PE 膜与顶部黄色美纹胶永久连接、不可分离，施工必须先按压黄色胶，再向下拉膜。

### 第二步：拆解参考成片

```bash
REPO="$PWD"
PROJECT="$REPO/03_视频项目/防尘膜短视频"

python3 "$REPO/90_视频生产工具/video-content-skills/skills/reference-video-analyzer/scripts/analyze_reference.py" \
  --input "$PROJECT/01_原始资料/reference_videos/reference-01.mp4" \
  --output-dir "$PROJECT/02_参考片拆解/reference-01"
```

输入哈希未变化时会复用分析，不应重复消耗时间。

### 第三步：编译并检查变体

```bash
FACTORY="$REPO/90_视频生产工具/video-content-skills/skills/reference-ad-factory"

python3 "$FACTORY/scripts/compile_variants.py" --project "$PROJECT" --check-only
python3 "$FACTORY/scripts/compile_variants.py" --project "$PROJECT" --force
```

缺少真实脚本、产品约束、素材来源或出现不支持的产品声称时，应停止而不是继续生成。

### 第四步：预览付费差量

```bash
PRODUCER="$REPO/90_视频生产工具/video-content-skills/skills/video-batch-producer"
GEN_PROJECT="$PROJECT/04_AI生成工程/你的批次目录"

python3 "$PRODUCER/scripts/run_pipeline.py" \
  --project "$GEN_PROJECT" --max-workers 2 --retries 0
```

记录 dry-run 输出中的图片任务数、视频任务数、付费秒数、模型和估算费用。任何数字与预期不符都不要执行。

### 第五步：只生成关键帧

```bash
python3 "$PRODUCER/scripts/run_pipeline.py" \
  --project "$GEN_PROJECT" \
  --execute --cost-authorized --keyframes-only \
  --max-image-jobs 3 --max-workers 2 --retries 0
```

`3` 只是示例上限，必须替换为刚刚批准的实际数量。逐张查看原分辨率图片，并使用 `review_keyframe.py` 写入批准或拒绝记录。没有合格图片，不得进入下一步。

### 第六步：生成视频

```bash
python3 "$PRODUCER/scripts/run_pipeline.py" \
  --project "$GEN_PROJECT" \
  --execute --cost-authorized \
  --max-image-jobs 0 --max-video-jobs 3 --max-paid-video-seconds 30 \
  --max-workers 2 --retries 0
```

同样地，`3` 和 `30` 只是三条 10 秒素材的示例。nanyao 需要公网可访问的关键帧 URL；本地路径、localhost、内网地址和 base64 不能作为输入。

### 第七步：验收、组装和发布前检查

```bash
python3 "$PRODUCER/scripts/verify_project.py" --project "$GEN_PROJECT"
```

技术通过后仍需人工检查 1、3、5、7、9 秒画面。正式广告再加入独立旁白、字幕和后期，最终用 `assemble_variants.py` 先 dry-run、再组装。正式成片要求不少于 20 秒，通常不超过 60 秒；单个 10 秒场景素材不受最终成片时长约束。

## 8. 现场怎么展示

### 8.1 重要前提：GitHub 默认没有 MP4

为了保持仓库轻量，MP4/MOV、音频和 PPTX 默认被 `.gitignore` 排除。GitHub 保存的是更关键、可审计和可复用的内容：代码、Skill、脚本、提示词、参数、关键帧、对比图、哈希验收和报告。

因此：

- 只拿 GitHub 克隆：可以完整讲解架构、查看图片证据、运行测试和 dry-run；
- 要播放历史成片：负责人需要另行把本机 `06_成片/` 和场景目录下的 `案例视频/` 复制给演示同事；
- 不建议为了演示临时把大视频提交到 GitHub。

### 8.2 推荐的 8 分钟演示顺序

1. **1 分钟：打开根 README。** 先说清这是一套 Codex-native Skills 流水线，不是拖拽工具，也不是 Web 产品。
2. **1 分钟：打开参考片拆解。** 展示 `02_参考片拆解/reference-01/contact-sheet-full-1s.jpg` 和 `reference-breakdown.md`，说明系统怎样把成片变成结构证据。
3. **1 分钟：打开脚本变体。** 展示 Phase 2 v2 的五种策略，说明“同一参考片如何裂变”。
4. **2 分钟：打开场景总表和 Excel 台账。** 讲七维组合、一条提示词对应一条视频，以及为什么不能靠同一个提示词反复抽卡。
5. **1 分钟：打开关键帧与哈希记录。** 展示通过与拒绝案例，强调图片不过门就不花视频钱。
6. **1 分钟：打开五节点对比图。** 说明 API 成功不等于业务合格，系统检查 10 秒内结构是否漂移。
7. **1 分钟：运行单元测试或 dry-run。** 展示费用上限、无凭证零成本检查和可恢复执行。

如果已经拿到本地视频包，可在第 3 步播放一条 Phase 2 v2 成片，在第 6 步播放“通过”和“淘汰”素材各一条。不要只放最好看的视频；展示系统如何识别废片更能体现工程价值。

## 9. 对外讲解话术

### 30 秒版本

> 我们做的不是单次 AI 生视频，而是一套可复用的产品短视频生产 SOP。它先拆解参考片，再把产品事实、脚本策略和七维场景组合变成独立提示词；图片必须人工验收并绑定哈希，合格后才能调用视频模型；视频再做多节点验收、旁白字幕和后期。整个过程有费用硬上限、断点续跑和失败留档，所以换一个开发者或 Agent，也能沿着同一套规则继续生产和复盘。

### 2 分钟版本的四个亮点

1. **Skills 而不是拖拽封装。** 每个能力可独立触发、可读、可测试、可版本管理，也能由不同 Agent 复用。
2. **参考驱动但事实隔离。** 学习爆款视频的信息密度和节奏，同时用产品事实约束模型，避免把参考片里的错误结构带进新广告。
3. **质量门前置。** 先审便宜的图片，再生成昂贵的视频；哈希把验收结论绑定到具体文件，替换素材后不能沿用旧批准。
4. **批量但不重复。** 用七维策略和独立提示词制造受控差异，同时保留失败样本和实际通过率，不用“生成成功率”冒充“业务合格率”。

## 10. 常见问题与标准回答

### 这是一个“一键自动出片”项目吗？

不是完全无人值守。确定性步骤、API 调用、断点续跑、费用限制和留档已经自动化；产品事实、图片结构、视频语义和最终广告效果必须有人验收。这是刻意设计的质量边界。

### 为什么还需要先生成图片？

当前稳定主链路是图生视频。图片可以低成本锁定产品结构、构图、人物和场景；图片不合格就拦截，比直接让视频模型连续十秒自由发挥更可控、更省钱。

### 为什么不直接使用模型自带口播？

当前 nanyao Grok 文档不保证可控的原生对白。正式广告使用独立 TTS 和字幕路线，才能确定台词、语速、声音角色和时间轴。模型自带声音可以作为素材，但不能当作稳定交付承诺。

### 为什么 GitHub 看不到成片？

这是有意的轻量化策略。大媒体文件不适合跟随每次代码提交；GitHub 保存可复刻流程和证据，视频由本地交付目录或独立对象存储管理。

### 能否换产品或换模型？

可以。产品事实、参考片和七维场景是业务输入；图片与视频供应商封装在生产适配器中。换产品主要新增业务项目和策略台账，换模型主要修改适配器及模型规则，但都必须重新验证参数、价格和质量门。

### 当前通过率是不是 100%？

不是，也不应这样宣称。当前不同场景批次出现过 2/3、3/3 和 6/7。项目亮点正是如实记录废片率，并在昂贵环节之前尽可能拦截错误。

### 502、522、525 是否等于生成失败？

不一定。它可能发生在轮询或下载阶段，而供应商任务已经完成。必须先根据任务 ID 查询原任务和成片 URL；确认没有已存在任务后，才能考虑重新提交。

## 12. 接手后的开发规范

- 一个可验证的小变更对应一次 commit + push；
- 只提交本次修改，不覆盖其他人的未提交文件；
- 修改代码、参数或入口命令时，同步更新 README/AGENTS，不能叠加冲突说明；
- 面向业务人员的文档、表格、字段和提示词尽量使用中文；
- 大视频、音频、PPT 和凭证不进 Git；
- `.work/` 是唯一临时区，不在业务目录创建额外临时文件夹；
- 生产台账只记录真实执行过的路线和实际提示词；
- 提交前运行测试和 `check_repository.py`；
- 任何付费批次先预览、再授权、再设硬上限；
- 不删除失败证据，把明确废片放入垃圾桶用于回顾。
